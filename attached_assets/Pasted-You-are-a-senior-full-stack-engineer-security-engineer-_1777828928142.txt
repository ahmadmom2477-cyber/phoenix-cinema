Very important free trial requirement:
Each user must get exactly 3 free plays/views at the beginning.

Correct behavior:
- A new user can watch/play up to 3 times for free.
- If the user uses 1 free play, then clicks the subscription button or opens the subscription page, they must still be able to go back and continue using the remaining 2 free plays.
- Do NOT force subscription immediately just because the user visited or clicked the subscription page.
- Subscription should only become required after the user has used all 3 free plays.
- If the user has 0 free plays remaining and no active subscription, then they must be blocked from watching and must subscribe.
- The frontend should clearly show how many free plays remain.
- Example:
  - User starts with 3 free plays.
  - User watches 1 item.
  - Remaining free plays = 2.
  - User clicks Subscribe.
  - User closes or goes back.
  - User can still watch 2 more free plays.
  - After the third play is used, watching becomes locked behind subscription.

Do not rely on localStorage as the source of truth for free plays or subscription.

Implement free trial properly on the server side.

Recommended free trial design:
- When a new visitor opens the app, create or retrieve an anonymous trial session.
- Store the trial session server-side in **secure environment variables**.
- Use a secure httpOnly cookie or secure server-issued trial token.
- Track:
  - trialId
  - createdAt
  - freePlaysUsed
  - freePlaysLimit = 3
  - freePlaysRemaining
  - lastUsedAt
  - optional device/session metadata
- The server must decide whether the user can watch.
- The client can display cached trial info, but must always verify with the server.
- Clearing localStorage must not reset the trial.
- Visiting the subscription page must not consume a free play.
- Clicking the subscription button must not consume a free play.
- Only an actual watch/play action should consume a free play.
- Avoid double-counting reloads or accidental repeated API calls consuming all free plays.
- Add a short idempotency window or playback session ID so refreshes do not instantly burn multiple plays.
- After free plays are exhausted, protected watch/media/source routes must return 403 or 402 with a clear JSON response saying subscription is required.

Subscription requirement:
- Subscription must be validated server-side.
- localStorage may only be used for UI convenience, never as the source of truth.
- When a user activates a subscription, gift code, or access code, the server must create a secure subscription session.
- The subscription session should include:
  - token
  - createdAt
  - expiresAt
  - plan/type
  - optional device/session metadata
- Store subscription sessions server-side in **environment variables**.
- /api/subscription/check-token must verify the token against server-side data.
- Never trust expiration dates sent by the client.
- Expired or invalid tokens must be rejected.
- Use secure random token generation.
- Frontend requests to protected premium routes must include:
  Authorization: Bearer <token>
- If no valid subscription exists but the user has free plays remaining, allow free playback and decrement the free play count correctly.
- If no valid subscription exists and free plays are exhausted, block playback.

Access rules:

1. They have an active valid subscription session, OR
2. They still have free plays remaining.

A user must be blocked if:
1. They have no active subscription, AND
2. Their 3 free plays are fully used.
